概述#

如果你已经有一定工程经验，Docker 最有价值的点，不是“会几个命令”，而是把项目运行环境、依赖和服务编排变成一套可重复执行的工程描述。

这篇文章不打算把 Docker 的全部概念讲一遍，而是围绕一个完整的本地项目场景，串起从入门到实践最常用的部分：

  • 用 Docker 跑一个 Nginx 静态站点
  • 用 Docker 构建一个 Node.js 或 Python/FastAPI 应用
  • 用 Docker Compose 编排 MySQL + Redis + App
  • 理解镜像、容器、端口、卷、网络这些真正会影响日常开发的核心概念

先说结论：

  • 镜像是“运行模板”，容器是“运行中的实例”
  • docker run 适合单服务验证，docker compose 更适合真实项目
  • 真正落地时，最容易出问题的不是“命令不会写”，而是端口、卷、网络、启动顺序和环境变量
  • 对工程项目来说，Compose 文件和 Dockerfile 本身就是运行文档

Docker 到底解决什么问题#

对有基础的工程师来说，Docker 的价值主要有三点：

1. 环境一致性#

本地能跑、测试能跑、线上也能跑，依赖的是同一套镜像和配置，而不是“我电脑上正好装过”。

2. 服务隔离#

MySQL、Redis、App 不需要都直接装在宿主机上。每个服务跑在自己的容器里，依赖边界更清晰。

3. 低成本复现#

拉代码、执行命令、启动编排，整个项目环境就能快速恢复。对多人协作、本地调试、临时排障都很有用。

先理解 4 个最核心概念#

1. 镜像#

镜像可以理解为容器的只读模板，里面包含应用代码、运行时、依赖和启动命令。

常见操作：

docker pull nginx:latest
docker images

2. 容器#

容器是镜像启动后的运行实例。一个镜像可以启动多个容器。

常见操作：

docker ps
docker ps -a
docker logs <container_name>
docker exec -it <container_name> sh
docker rm -f <container_name>

3. 端口映射#

容器内部端口默认不能直接从宿主机访问，需要用 -p 映射出来。

docker run -d --name web -p 8080:80 nginx:latest

这里的含义是：把宿主机 8080 映射到容器内的 80。

4. 数据卷#

容器删掉后，容器内部数据可能一起消失。数据库、日志、上传文件这类需要持久化的数据，通常要挂载卷。

docker run -d --name mysql \
  -v mysql_data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=123456 \
  mysql:8

如何理解 Dockerfile 与 compose.yaml#

可以把两者这样理解：

  • Dockerfile 解决的是“一个服务怎么被做出来”
  • compose.yaml 解决的是“多个服务怎么一起跑起来”

更具体一点：

1. Dockerfile 是镜像构建说明书#

它定义基础镜像、工作目录、依赖安装、代码拷贝、暴露端口、启动命令，最终产出的是一个镜像。

典型命令：

docker build -t my_app:latest .

2. compose.yaml 是运行编排说明书#

它定义有哪些服务、每个服务用哪个镜像或哪个 Dockerfile 构建、端口怎么映射、环境变量怎么传、卷怎么挂、服务之间怎么连，最终产出的是一组一起协作的容器。

典型命令：

docker compose up -d

3. 两者在项目里的关系#

一个常见项目里，app 服务会有自己的 Dockerfile，而 compose.yaml 负责把 app、mysql、redis 等服务组织起来。

services:
  app:
    build: .
  mysql:
    image: mysql:8
  redis:
    image: redis:7-alpine

这里的含义是：

  • app 使用当前目录里的 Dockerfile 构建
  • mysql 和 redis 直接使用现成镜像
  • 最后由 Compose 统一启动和管理

如果只记一句话，可以记成：

  • Dockerfile 关注“怎么构建单个服务镜像”
  • compose.yaml 关注“怎么组织整个项目运行”

从一个 Nginx 静态站点开始#

先从最简单的场景入手。假设当前目录下有一个 index.html：

<!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8" />
  <title>Docker Demo</title>
</head>
<body>
  <h1>Hello Docker</h1>
</body>
</html>

再写一个最小 Dockerfile：

FROM nginx:alpine
COPY . /usr/share/nginx/html

构建并运行：

docker build -t demo-nginx:latest .
docker run --rm -p 8080:80 demo-nginx:latest

浏览器访问 http://localhost:8080，能看到页面就说明整条链路已经通了：

  • Dockerfile 定义镜像
  • docker build 构建镜像
  • docker run 启动容器
  • -p 暴露服务端口

这是 Docker 最小可用闭环。

应用服务如何容器化#

真实项目里，通常不只是静态页面，还会有应用服务。下面分别看 Node.js 和 FastAPI 两种常见写法。

1. Node.js 应用#

一个最小 Node.js Dockerfile 可以这样写：

FROM node:20-alpine

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

EXPOSE 3000
CMD ["npm", "run", "start"]

构建和运行：

docker build -t demo-node:latest .
docker run --rm -p 3000:3000 demo-node:latest

2. Python / FastAPI 应用#

一个最小 FastAPI Dockerfile：

FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

构建和运行：

docker build -t demo-fastapi:latest .
docker run --rm -p 8000:8000 demo-fastapi:latest

这里有一个关键点：应用必须监听 0.0.0.0，不能只监听 127.0.0.1。否则容器外部访问不到。

为什么真实项目更适合 Docker Compose#

单个服务可以直接用 docker run，但一旦项目变成：

  • 一个 App
  • 一个 MySQL
  • 一个 Redis
  • 可能还有 Nginx 反向代理

继续手写多个 docker run 就会迅速失控。这个时候应该交给 docker compose。

Compose 的作用，不只是“批量启动容器”，更重要的是用一个 YAML 文件把服务关系、环境变量、端口、挂载、依赖都描述清楚。

一个最小项目实战：MySQL + Redis + FastAPI#

下面给一个典型本地开发编排示例：

version: "3.9"

services:
  app:
    build: .
    container_name: demo_app
    ports:
      - "8000:8000"
    environment:
      MYSQL_HOST: mysql
      MYSQL_PORT: 3306
      MYSQL_USER: root
      MYSQL_PASSWORD: 123456
      MYSQL_DB: demo
      REDIS_HOST: redis
      REDIS_PORT: 6379
    depends_on:
      - mysql
      - redis

  mysql:
    image: mysql:8
    container_name: demo_mysql
    ports:
      - "3306:3306"
    environment:
      MYSQL_ROOT_PASSWORD: 123456
      MYSQL_DATABASE: demo
    volumes:
      - mysql_data:/var/lib/mysql

  redis:
    image: redis:7-alpine
    container_name: demo_redis
    ports:
      - "6379:6379"

volumes:
  mysql_data:

启动命令：

docker compose up -d
docker compose ps
docker compose logs -f

停止并清理：

docker compose down

如果连卷也一起清理：

docker compose down -v

这个示例里有几个很关键的实践点：

1. 服务之间通过服务名通信#

应用连接 MySQL 时，不应该写 localhost，而应该写 mysql；连接 Redis 时写 redis。

因为在 Compose 网络里，服务名本身就是可解析的主机名。

2. depends_on 不等于“服务已就绪”#

depends_on 只能保证启动顺序，不能保证 MySQL 已经初始化完成。

所以 App 如果在启动时立刻连数据库，仍然可能失败。更稳妥的方式是：

  • App 里自己做重试
  • 或增加健康检查与等待逻辑

这是很多人第一次用 Compose 时最容易误解的点。

3. 数据库一定要考虑持久化#

MySQL 这种服务如果不挂卷，容器删掉后数据就没了。

volumes:
  - mysql_data:/var/lib/mysql

这不是“可选优化”，而是本地开发最基本的保护。

Nginx 在项目里怎么用#

Nginx 在 Docker 场景里常见有两种角色：

1. 静态资源服务#

前端打包产物直接交给 Nginx 托管，适合纯静态站点或前后端分离项目中的静态资源层。

2. 反向代理入口#

把 /api 转发给 App，把 / 指向静态资源。这样本地环境更接近真实部署结构。

一个最小 nginx.conf 示例：

server {
    listen 80;

    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://app:8000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

这里 proxy_pass 指向的是 Compose 里的服务名 app，不是宿主机地址。

日常最常用的命令#

对工程项目来说，下面这些命令最常用：

docker ps
docker logs -f <container_name>
docker exec -it <container_name> sh
docker images
docker build -t <image_name> .
docker run --rm -p 8080:80 <image_name>
docker compose up -d
docker compose down
docker compose ps
docker compose logs -f app

如果只记一套核心命令，这些基本够覆盖大多数本地开发场景。

常见坑点#

1. 宿主机端口冲突#

最常见的问题不是容器没起来，而是端口早就被占了。

比如本机已经装了 MySQL，再执行：

docker compose up -d

如果 Compose 里也映射了 3306:3306，就会直接冲突。

2. 容器里不能用 localhost 连其他服务#

在容器里，localhost 指向的是容器自己，不是宿主机，也不是 Compose 里的其他服务。

所以：

  • App 连 MySQL 应写 mysql
  • App 连 Redis 应写 redis

3. 应用监听地址写错#

很多 Web 服务默认只监听 127.0.0.1。在容器里这通常不够，应该监听 0.0.0.0。

否则端口虽然映射了，外面依然访问不到。

4. 只会重启，不会看日志#

容器启动失败时，第一反应不应该是不断 up 和 restart，而应该先看日志：

docker logs <container_name>
docker compose logs -f app

Docker 排障的第一入口，通常就是日志。

5. 把容器当虚拟机用#

容器不是给你长期手工改环境的。正确思路是：

  • 改 Dockerfile
  • 改 Compose 配置
  • 重新构建
  • 重新启动

如果每次都靠 exec 进容器手改，最后环境一定不可复现。

小结#

对有基础的工程师来说，Docker 真正需要掌握的不是“命令大全”，而是三件事：

  • 知道镜像、容器、端口、卷分别解决什么问题
  • 知道单服务用 docker run，多服务用 docker compose
  • 知道真实项目里最容易出问题的是网络、持久化、启动时序和配置管理

如果你只是想快速验证一个服务，docker build 加 docker run 就够了。 如果你在跑一个完整项目，尽早把环境描述收敛到 Dockerfile + compose.yaml，后续维护成本会低很多。
