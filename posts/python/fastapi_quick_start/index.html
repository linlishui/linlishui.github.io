FastAPI快速入门：写第一个REST API

April 13, 2026 · 2 min · 375 words · 夏天的小西瓜
Table of Contents

如果你现在学 Python 后端，目标多半不是做传统服务端页面，而是写接口、接数据库、加中间件，给前端或 AI 服务提供 API。这个方向下，FastAPI 比 Flask 更贴近现代开发习惯。

这篇文章不展开讲框架历史，重点只放在实战入门：

  • 写第一个 REST API
  • 理解中间件和数据库接入入口
  • 搞清楚状态码和接口命名规范

先跑起来#

安装依赖：

pip install fastapi uvicorn

新建 main.py：

from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def read_root():
    return {"message": "hello"}

运行：

uvicorn main:app --reload

启动后访问：

  • http://127.0.0.1:8000/
  • http://127.0.0.1:8000/docs

/docs 是 FastAPI 很实用的一点。接口定义完，文档就能直接看和调试。

写第一个 REST API#

先用一个 items 资源做例子：

from fastapi import FastAPI, HTTPException, status
from pydantic import BaseModel

app = FastAPI()

# 用内存字典模拟数据库，方便先理解接口结构
items_db = {
    1: {"id": 1, "name": "keyboard", "price": 199.0}
}

class ItemIn(BaseModel):
    name: str
    price: float

@app.get("/items")
def list_items():
    # 查询资源列表
    return list(items_db.values())

@app.get("/items/{item_id}")
def get_item(item_id: int):
    # 按资源 ID 查询单个对象
    item = items_db.get(item_id)
    if item is None:
        raise HTTPException(status_code=404, detail="item not found")
    return item

@app.post("/items", status_code=status.HTTP_201_CREATED)
def create_item(item: ItemIn):
    # 创建资源时生成新 ID，并返回 201
    new_id = max(items_db.keys(), default=0) + 1
    new_item = {"id": new_id, **item.model_dump()}
    items_db[new_id] = new_item
    return new_item

@app.put("/items/{item_id}")
def update_item(item_id: int, item: ItemIn):
    # PUT 表示整体更新，目标资源不存在时返回 404
    if item_id not in items_db:
        raise HTTPException(status_code=404, detail="item not found")
    items_db[item_id] = {"id": item_id, **item.model_dump()}
    return items_db[item_id]

@app.delete("/items/{item_id}", status_code=status.HTTP_204_NO_CONTENT)
def delete_item(item_id: int):
    # 删除成功时返回 204，一般不再返回响应体
    if item_id not in items_db:
        raise HTTPException(status_code=404, detail="item not found")
    del items_db[item_id]

这段代码已经覆盖了最常见的 REST 操作：

  • GET /items：查询列表
  • GET /items/{item_id}：查询单个资源
  • POST /items：创建资源
  • PUT /items/{item_id}：整体更新资源
  • DELETE /items/{item_id}：删除资源

状态码要返回对#

后端接口不能只“能跑”，还要把语义返回正确。

最常见的状态码可以先记住这些：

  • 200 OK：请求成功，常用于查询和更新
  • 201 Created：创建成功，常用于 POST
  • 400 Bad Request：请求参数有问题
  • 401 Unauthorized：未登录或认证失败
  • 403 Forbidden：已登录，但没有权限
  • 404 Not Found：资源不存在
  • 500 Internal Server Error：服务端内部错误

在 FastAPI 里，常见做法是用 HTTPException：

raise HTTPException(status_code=404, detail="item not found")

如果是创建接口，可以直接声明：

@app.post("/items", status_code=201)

前端、客户端和调用方通常就是靠状态码判断请求结果，所以这不是小细节，而是接口契约的一部分。

接口命名尽量遵守 RESTful 风格#

刚开始写接口时，最容易乱的是命名。建议优先遵守这几个规则：

  • 路径里用名词，不用动词
  • 集合用复数，比如 /items
  • 单个资源用 /items/{item_id}
  • 用 HTTP 方法表达动作，不在路径里写 get、create、delete

例如：

  • 好：GET /items
  • 好：POST /items
  • 好：PUT /items/1
  • 差：GET /getItems
  • 差：POST /createItem
  • 差：POST /deleteItem

这样设计的好处是接口更统一，也更容易维护文档、权限和客户端调用逻辑。

中间件机制怎么理解#

中间件可以理解成“请求进入路由前、响应返回客户端前”的一层通用处理逻辑。

常见用途包括：

  • CORS
  • 请求日志
  • 鉴权
  • 统计耗时
  • 统一异常处理

例如加一个简单的耗时中间件：

import time
from fastapi import FastAPI, Request

app = FastAPI()

@app.middleware("http")
async def log_process_time(request: Request, call_next):
    start = time.time()
    # 继续把请求交给后面的路由或中间件处理
    response = await call_next(request)
    process_time = time.time() - start
    # 在响应头里附加处理耗时，便于调试
    response.headers["X-Process-Time"] = str(process_time)
    return response

理解中间件时，只要先抓住这条主线：

  • 请求先进入中间件
  • 中间件决定是否继续往后传
  • 路由处理完成后，再回到中间件补充响应

如果是前后端分离项目，CORS 中间件通常也很常见：

from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

数据库怎么接进来#

FastAPI 本身不负责数据库，但它很适合作为数据库访问的接口层。

常见组合是：

  • FastAPI：路由和接口层
  • Pydantic：请求和响应模型
  • SQLAlchemy：ORM
  • MySQL / PostgreSQL / SQLite：数据库

入门阶段不用一上来就写完整 ORM 层，但至少要知道结构上应该分开：

  • 路由层：接收请求和返回响应
  • 业务层：处理业务逻辑
  • 数据层：负责数据库读写

例如：

app/
  main.py
  routers/
    items.py
  schemas/
    item.py
  models/
    item.py
  db/
    session.py

最容易犯的错，是把数据库操作直接写死在路由函数里。代码能跑，但后续一接鉴权、日志、事务和缓存，很快就会乱。

小结#

FastAPI 入门最值得先掌握的，不是“高级特性”，而是这几件实战基础：

  • 用 GET / POST / PUT / DELETE 写出第一个 REST API
  • 理解中间件是在请求和响应之间做统一处理
  • 知道数据库要分层接入，不要把逻辑全堆在路由里
  • 返回正确状态码
  • 按 RESTful 风格设计接口路径

把这些打稳之后，再继续往认证、权限、异步任务和部署走，会顺很多。
