Android CLI 的重点不是“又多了一个 adb 替代品”，而是 Google 正在把 Android 工程能力整理成更适合 AI Agent 调用的命令入口。它把 SDK 管理、项目模板、模拟器、部署、UI 观察、截图定位、官方文档检索、Android skills、Android Studio 语义能力放到统一的 android 命令下。

如果只做手工开发，Android Studio、Gradle 和 adb 已经能覆盖绝大多数工作。但 AI 工程的核心问题不是“人能不能做”，而是“Agent 能不能稳定观察、执行、验证，并留下可复盘证据”。Android CLI 的价值就在这里：它把原本分散在 IDE、终端、设备和文档里的动作，收敛成一组更清晰的命令协议。

Android CLI 可以按下面这几层理解。表格比关系图更适合放在文章开头，读者可以先建立全局印象，再按后文逐段展开。

层级 能力 常用命令 Agent 拿到什么
环境与 SDK 确认 CLI、SDK 和工具链状态 android info、android sdk list/install/update 当前 SDK 路径、可用平台包、工具链是否完整
工程创建与理解 创建模板工程，分析现有项目 android create、android describe、./gradlew tasks module、构建目标、APK 产物路径
设备与运行 管理模拟器并部署应用 android emulator list/start/stop/create、android run 可用设备、安装结果、运行入口
观察与验证 读取 UI 树、截图、定位坐标 android layout、android screen capture/resolve、adb input 页面结构、可点击元素、视觉证据、交互坐标
知识与实践 查询官方资料，安装任务技能 android docs search/fetch、android skills 最新 API 建议、迁移指引、任务级操作规范
IDE 语义能力 复用 Android Studio 索引和预览 android studio analyze-file/find-usages/render-compose-preview 声明位置、引用点、静态分析、Compose Preview 结果

工具边界#

1. Android CLI 不是什么#

Android CLI 不是要替代所有 Android 工具。更准确地说，它是 Agent 友好的编排入口。

工具 主要职责 适合谁调用
android 环境、SDK、模拟器、部署、UI 观察、文档、skills、IDE 语义能力 Agent、脚本、开发者
./gradlew 编译、单测、集成测试、lint、打包 CI、开发者、Agent
adb 低层设备通信、日志、shell、输入事件、文件传输 开发者、Agent
Android Studio 编辑、调试、预览、索引、代码语义分析 开发者，也可被 android studio 子命令调用

实际工程里不要强行把所有事都塞给 android。更稳的分工是：

  • 用 android info 和 android describe 让 Agent 先理解环境和项目。
  • 用 Gradle Wrapper 做构建、测试、lint 和产物生成。
  • 用 android run 做部署。
  • 用 android layout、android screen 和 adb 输入命令做 UI 验证。
  • 用 android docs 和 android skills 降低 Agent 使用旧知识的概率。

2. 它解决的核心问题#

AI Agent 做 Android 任务时最常见的失败不是“不会写代码”，而是缺少稳定上下文：

  • 不知道当前 SDK 在哪里。
  • 不知道项目有哪些 module。
  • 不知道 APK 最终产物路径。
  • 不知道设备上当前到底显示什么。
  • 不知道某个 API 的最新推荐写法。
  • 只能看截图，无法得到 UI 树和可点击坐标。

Android CLI 把这些上下文变成命令输出。对 Agent 来说，这比让模型凭经验猜更可靠。

安装与初始化#

1. 安装方式#

以 macOS 和 Linux 为例，可以使用官方安装脚本。公司机器或 CI 环境建议先审阅脚本内容，再执行。

# macOS Apple Silicon
curl -fsSL https://dl.google.com/android/cli/latest/darwin_arm64/install.sh | bash

# macOS Intel
curl -fsSL https://dl.google.com/android/cli/latest/darwin_x86_64/install.sh | bash

# Linux x86_64
curl -fsSL https://dl.google.com/android/cli/latest/linux_x86_64/install.sh | bash

安装后先做最小验证：

command -v android
android --version
android info

如果希望 Agent 能识别 Android CLI 的工作方式，再执行：

android init

android init 会安装 android-cli skill，帮助支持 skills 的 Agent 理解 Android CLI 的命令边界。

2. 配置 SDK 路径#

一台机器上可能有多个 Android SDK。临时切换可以用 --sdk：

android --sdk=/Users/me/Library/Android/sdk info
android --sdk=/opt/android-sdk sdk list --all

长期固定可以写入 ~/.androidrc：

--sdk=/Users/me/Library/Android/sdk

这比在多个脚本里到处改 ANDROID_HOME 更适合 Agent 执行，因为命令入口更明确。

3. 安装检查清单#

第一次接入 Android CLI，建议按下面的顺序检查：

检查项 命令 期望结果
CLI 是否在 PATH 中 command -v android 能输出可执行文件路径
CLI 版本 android --version 能输出版本号
SDK 路径 android info 能看到 sdk 字段
SDK 包可见 android sdk list --all 能列出本地和可安装包
模拟器可见 android emulator list 能列出 AVD，或者明确为空
Agent skill 初始化 android init 能完成初始化

如果这些命令里任何一步失败，不建议马上让 Agent 改业务代码。先把环境问题解决，否则后续错误很容易被误判成代码问题。

基础使用#

1. SDK 管理#

常用命令如下：

android sdk list --all
android sdk install platforms/android-36 build-tools/36.0.0
android sdk update
android sdk remove build-tools/35.0.0

本地开发可以用 android sdk update 保持工具链新鲜；普通 CI 不建议这么做。CI 更应该固定 Android SDK、Build Tools、AGP、Gradle Wrapper、JDK 版本，否则构建结果会随工具链变化而漂移。

更推荐在项目里显式记录依赖工具链：

docs/android-toolchain.md
gradle/wrapper/gradle-wrapper.properties
settings.gradle.kts
build.gradle.kts
.github/workflows/android.yml

Agent 接手任务时，先读这些文件，再决定要不要安装额外 SDK。

2. 创建项目#

查看模板：

android create --list --name=Probe

创建最小工程：

android create empty-activity \
  --name="CliProbe" \
  --output=./cli_probe

Android CLI 仍在快速迭代，不同版本的模板可能有细节差异。实际使用前建议先执行：

android create -h

本机验证时默认模板是 empty-activity，标签包含 compose、activity、agp-9。它适合让 Agent 快速创建一个最小 Compose 工程，用来复现 API、UI 或构建问题。

3. 理解现有工程#

在已有项目中，推荐先执行：

android info
android describe --project_dir=.
./gradlew tasks

android describe 会分析工程结构，并输出构建目标和产物路径等元信息。对 Agent 来说，这比猜 app/build/outputs/... 靠谱。

如果项目很大，可以让 Agent 先回答三个问题：

  • 哪些 module 会产出 APK 或 AAB？
  • 当前任务最可能影响哪个 module？
  • 修改后应该运行哪个 Gradle task 验证？

这一步能显著减少“全仓库乱搜”和“构建了错误 module”的情况。

4. 模拟器管理#

查看已有 AVD：

android emulator list

查看可创建的设备 profile：

android emulator create --list-profiles

启动和停止模拟器：

android emulator start Pixel_6_API_35
android emulator start --cold Pixel_6_API_35
android emulator stop Pixel_6_API_35

android emulator start 会等到模拟器完全启动并可用后再返回，这对自动化脚本很关键。传统脚本经常需要手写 adb wait-for-device 和额外 sleep，Android CLI 在这里提供了更清晰的同步点。

5. 构建和部署#

android run 只负责部署，不负责构建。完整链路通常是：

./gradlew :app:assembleDebug

android run \
  --apks=app/build/outputs/apk/debug/app-debug.apk \
  --device=emulator-5554

如果一个产物包含多个 APK，可以用逗号传入：

android run \
  --apks=base.apk,config.arm64_v8a.apk,config.xxhdpi.apk \
  --device=emulator-5554

多设备连接时必须指定 --device。不要让 Agent 自己猜目标设备。

高频场景#

1. 从零创建一个复现工程#

适合验证某个 API、Compose 组件、构建插件或平台行为：

android create empty-activity --name="ApiProbe" --output=./api_probe
cd api_probe
android info
./gradlew :app:assembleDebug
android emulator start Pixel_6_API_35
android run --apks=app/build/outputs/apk/debug/app-debug.apk

这个流程的优势是干净。不要在大项目里直接验证所有猜想，先用最小工程排除工具链和 API 理解问题。

2. 接手一个陌生 Android 项目#

推荐让 Agent 先按这个顺序工作：

android info
android describe --project_dir=.
./gradlew projects
./gradlew tasks

然后再让它阅读：

  • settings.gradle 或 settings.gradle.kts
  • 根目录 build.gradle 或 build.gradle.kts
  • 目标 module 的 build.gradle
  • AndroidManifest.xml
  • 入口 Activity 或 Compose navigation 入口

这比直接让 Agent 全局搜索“MainActivity”更可靠。大型项目里入口可能不止一个，构建产物也可能不在默认路径。

3. UI 改动后的验证#

修改 UI 后，不要只看构建是否通过。推荐最小验证闭环：

./gradlew :app:assembleDebug
android run --apks=app/build/outputs/apk/debug/app-debug.apk --device=emulator-5554
android layout --pretty --output=build/ai/layout.json
android screen capture --annotate --output=build/ai/screen.png

layout 适合判断结构，截图适合判断视觉。比如：

  • 文案是否出现在 UI 树里：看 layout.json。
  • 按钮是否可点击：看 interactions 是否包含 clickable。
  • 输入框是否聚焦：看 state 是否包含 focused。
  • 颜色、遮挡、图片资源是否正确：看截图。

如果第一次 layout 输出太大，后续操作后可以用 diff：

android layout --diff --pretty

这样能让 Agent 只关注变化部分，减少上下文噪音。

4. 截图标注和坐标定位#

当 UI 树找不到目标元素，或者页面是 WebView、动画、图片内容时，使用标注截图：

android screen capture --annotate --output=build/ai/screen.png

假设截图中目标按钮标号是 #5，可以转换成真实坐标：

android screen resolve \
  --screenshot=build/ai/screen.png \
  --string="input tap #5"

继续执行点击：

adb shell $(android screen resolve \
  --screenshot=build/ai/screen.png \
  --string="input tap #5")

这比让模型凭截图估算坐标更稳，也更容易复盘。

5. 用官方文档给 Agent 做 grounding#

AI 写 Android 代码的主要风险之一，是用旧知识解决新问题。android docs 把 Android Knowledge Base 接进终端，适合在生成代码前先确认最新推荐写法。

android docs search "Compose screenshot testing"
android docs fetch kb://android/studio/preview/compose-screenshot-testing

推荐在这些场景中强制先查文档：

  • Android 新版本行为变化。
  • AGP、Gradle、Kotlin 插件升级。
  • Compose、CameraX、Navigation、Media、Permissions 等 Jetpack API。
  • 权限、后台限制、前台服务、通知、存储等平台策略。
  • 官方明确有迁移指南的任务。

进阶使用#

1. 建立 Agent 工作协议#

可以把 Android Agent 工作流固定成七步：

1. android info：确认 SDK 和 CLI 版本
2. android describe：理解工程结构和产物路径
3. android docs search/fetch：查询相关官方实践
4. 修改代码：只改必要文件
5. ./gradlew assembleDebug 或相关 test task
6. android run + android layout/screen：部署并观察结果
7. 输出变更文件、验证命令、关键截图或 JSON 证据

这套协议的价值是减少“模型猜测”。每一步都有命令证据，失败也能定位在环境、构建、部署、UI 状态或代码实现。

更严格一点，可以要求 Agent 每次交付时给出四类结果：

结果 说明
变更文件 改了哪些 Kotlin、Java、XML、Gradle 或资源文件
验证命令 实际运行了哪些命令
关键证据 layout.json、截图、Gradle 输出、日志片段
残余风险 没跑的测试、未覆盖设备、仍需人工确认的视觉细节

2. 观察优先，而不是改完再猜#

Agent 调 UI 问题时，建议按“结构优先、视觉补充”的顺序：

android layout
  -> 如果能看到目标元素，用 text/resourceId/contentDesc/bounds 判断
android layout --diff
  -> 如果刚做过操作，用 diff 关注变化
android screen capture --annotate
  -> 如果 UI 树不够，转成截图标注
android screen resolve + adb shell input
  -> 需要交互时再把标号转成坐标

不要一上来就让 Agent 用 adb shell input tap 500 1200。这种坐标在不同分辨率、字体大小、系统导航模式下都容易失效。

3. 用 Journeys 描述核心用户路径#

Journeys 是面向 Agent 的自然语言测试说明。它不是传统 Espresso 脚本，而是把用户路径写成 XML，让 Agent 按步骤操作并验证。

<journey name="login success">
  <description>验证用户可以完成登录并进入首页</description>
  <actions>
    <action>Tap the username input</action>
    <action>Enter text "demo@example.com"</action>
    <action>Tap the password input</action>
    <action>Enter text "123456"</action>
    <action>Tap the "Login" button</action>
    <action>Verify that the Home screen is visible</action>
  </actions>
</journey>

写 Journey 时有几个经验：

  • 一个 <action> 只表达一个动作或一个验证点。
  • Tap、Enter text、Scroll 这类动作步骤应该能被 UI 明确执行。
  • Verify 或 Check 步骤只做观察，不应该隐含点击、滚动或输入。
  • 失败也是结果，不要在 Journey 里写“如果失败就尝试另一种方式”。
  • 核心路径先写短，稳定后再增加分支和异常场景。

适合写 Journey 的场景：

  • 登录、注册、搜索、下单这类主路径。
  • 多端形态适配后的关键页面检查。
  • 修复线上缺陷后的回归验证。
  • AI Agent 修改 UI 后的端到端冒烟验证。

Journey 的关键不是“自动化覆盖率”，而是把产品路径变成 Agent 可以执行和复盘的验收合同。

4. 借 Android Studio 做语义查询#

android studio 子命令可以连接运行中的 Android Studio，借 IDE 做语义分析。典型命令：

android studio check

android studio analyze-file \
  app/src/main/java/com/example/MainActivity.kt

android studio find-declaration --short HomeScreen
android studio find-usages --short HomeScreen

android studio render-compose-preview \
  --output-image-file=build/ai/home_preview.png \
  --print-semantics \
  app/src/main/java/com/example/HomeScreen.kt \
  HomeScreenPreview

这对 AI 工程很有用。Agent 不再只靠 rg 做文本搜索，而是可以借 IDE 解析 Kotlin/Java 符号、引用关系、lint 问题和 Compose Preview 语义树。

适合调用 Studio 能力的任务：

  • 同名函数或同名 Composable 很多，文本搜索容易误判。
  • 需要找真实声明、引用点或调用链。
  • 需要确认 Compose Preview 是否能渲染。
  • 想让 IDE 对单个文件做静态分析。

注意：studio 命令仍属于预览能力，要求项目在支持版本的 Android Studio 中打开，并启用 Gemini in Android Studio。它适合本地智能开发，不建议直接作为稳定 CI 基础。

5. 用 Android skills 固化团队经验#

android skills list --long
android skills find performance
android skills add edge-to-edge
android skills add agp-9-upgrade
android skills add --all

Android skills 可以把某类任务的最佳实践交给 Agent，例如：

  • XML View 迁移 Compose。
  • Edge-to-edge 适配。
  • AGP 9 升级。
  • R8 配置审计。
  • CameraX 迁移。
  • Navigation 3 接入。
  • 测试环境搭建。

如果团队要自定义 skill，一定要改名。官方提示过，直接修改已安装 skill，后续 skills add 更新时可能被覆盖。

团队内部可以把 skills 分成三类：

类型 内容 是否适合共享
官方 skill Google 提供的迁移、测试、性能、Compose 等实践 适合直接安装
团队规范 skill 架构分层、命名、日志、埋点、错误处理规范 适合内部共享
项目专用 skill 某个项目的 module 结构、发布流程、测试账号 只适合项目内使用

提效脚本#

1. 生成 AI 调试证据#

可以在项目中放一个 scripts/android_ai_probe.sh：

#!/usr/bin/env bash
set -euo pipefail

DEVICE="${ANDROID_SERIAL:-}"
APK="${APK:-app/build/outputs/apk/debug/app-debug.apk}"
OUT_DIR="${OUT_DIR:-build/ai-artifacts}"

mkdir -p "$OUT_DIR"

android info | tee "$OUT_DIR/android-info.txt"
android describe --project_dir=. > "$OUT_DIR/android-describe.txt" || true

./gradlew :app:assembleDebug | tee "$OUT_DIR/gradle-assemble.txt"

if [[ -n "$DEVICE" ]]; then
  android run --apks="$APK" --device="$DEVICE"
  android layout --device="$DEVICE" --pretty --output="$OUT_DIR/layout.json"
  adb -s "$DEVICE" exec-out screencap -p > "$OUT_DIR/screen.png"
else
  android run --apks="$APK"
  android layout --pretty --output="$OUT_DIR/layout.json"
  android screen capture --annotate --output="$OUT_DIR/screen.png"
fi

这个脚本的目的不是替代完整测试，而是给 Agent 留下运行现场：工程信息、APK、UI 树和截图。当前示例中 layout 显式支持 --device，多设备截图则用 adb -s 保证目标明确；单设备环境下继续使用 android screen capture --annotate 生成标注截图。后续排查“页面没跳转”“按钮不可点”“文本没显示”时，不需要重新猜状态。

2. UI 问题排查模板#

当用户说“这个页面不对”时，可以要求 Agent 按这个模板处理：

请按下面顺序排查：
1. 运行 android describe --project_dir=.，确认目标 module 和 APK 路径。
2. 构建目标 APK。
3. 用 android run 安装到指定设备。
4. 用 android layout --pretty 保存 UI 树。
5. 用 android screen capture --annotate 保存截图。
6. 根据 UI 树和截图判断问题属于数据、布局、样式、交互还是导航。
7. 修改最小必要代码。
8. 重新构建、部署、采集 layout 和截图。
9. 给出前后证据路径和剩余风险。

这类提示词比“帮我修一下页面”更有效，因为它约束了 Agent 的观察、修改和验证方式。

3. 适合放进 CI 的最小检查#

普通 CI 不一定要启动模拟器，但至少可以做：

android info
./gradlew :app:assembleDebug
./gradlew :app:testDebugUnitTest

如果 CI 有稳定模拟器环境，可以增加：

android emulator start --cold Pixel_6_API_35
android run --apks=app/build/outputs/apk/debug/app-debug.apk
android layout --pretty --output=build/ci/layout.json
android screen capture --output=build/ci/screen.png
android emulator stop Pixel_6_API_35

注意，涉及视觉截图的 CI 要控制设备 profile、系统版本、字体大小、主题、语言和动画设置，否则结果会不稳定。多设备 CI 还要显式传 --device 给支持该参数的命令；截图如果无法指定设备，优先收敛到单设备环境，或者改用 adb -s <serial> exec-out screencap -p。

常见坑#

1. HOME 不可写会导致命令失败#

Android CLI 启动时会写入本地配置和 analytics 目录。沙箱或 CI 中如果 $HOME 不可写，可能连 android --version 都失败。

export HOME="$PWD/.ci-home"
mkdir -p "$HOME"
android --version

2. android run 不会自动构建#

下面这个命令只负责部署已有 APK：

android run --apks=app/build/outputs/apk/debug/app-debug.apk

如果 APK 不存在，先执行：

./gradlew :app:assembleDebug

不要让 Agent 把“部署失败”误判成“代码编译失败”。

3. layout 和截图不能互相替代#

layout 适合结构化判断，截图适合视觉判断。颜色、遮挡、裁剪、图片资源、WebView 内容，仍然需要截图确认。

反过来，截图也不能替代 layout。只靠截图很难稳定判断 resourceId、可点击状态、焦点状态和滚动容器。

4. 多设备环境必须显式指定目标#

多设备场景要先看设备：

adb devices
android emulator list

然后显式指定：

android run --apks=app-debug.apk --device=emulator-5554
android layout --device=emulator-5554 --pretty

5. 不要把自动更新放进普通 CI#

android update 适合本地追新。普通 CI 要固定版本；只有专门验证工具链兼容性的流水线才适合定期更新。

6. Studio 子命令不等于无头 CI 能力#

android studio 相关命令依赖正在运行的 Android Studio 和 IDE 索引。它适合本地 Agent 辅助开发，不要默认放进无头 CI。

7. Windows 能力要单独校验#

官方已说明 Windows 上 android emulator 当前不可用，PowerShell 下载 Android CLI 也有已知限制。跨平台脚本要显式分支。

小结#

Android CLI 的深层价值，是把 Android 开发拆成一组 Agent 可调用的工程动作：查环境、识别工程、查官方文档、构建、部署、观察 UI、截图定位、执行 Journey、调用 IDE 语义能力。

对个人开发者来说，它减少工具切换；对 AI 工程来说，它把“模型凭经验猜”变成“模型基于命令观察、执行和验证”。未来 Android Agent 能不能稳定完成复杂任务，很大程度取决于我们是否把这些 CLI 能力沉淀成脚本、提示词、CI 流程和团队 skills。

如果只记住一条原则，就是：不要让 Agent 直接猜 Android 工程状态。先让它用 android info、android describe、android layout 和 android screen 观察，再让它修改和验证。
