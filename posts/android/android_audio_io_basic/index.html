Android音频路由入门：设备、模式、焦点与音量

Android 音频输入输出的核心不是“播放一个声音”或“录一段声音”这么简单，而是系统需要根据场景选择合适的输入设备、输出设备、音频模式、音频焦点和音量类型。

例如音乐播放通常走扬声器、耳机或蓝牙 A2DP；语音通话通常走听筒、免提、蓝牙通话链路或 USB 耳机；录音则需要选择内置麦克风、耳机麦克风或 USB 麦克风。理解这些概念后，再看 AudioTrack、AudioRecord、MediaPlayer、MediaRecorder 才不会混乱。

基本概念#

1. 输入和输出#

在 Android 音频系统中，设备通常可以分成两类：

  • 输入设备：给系统提供音频数据，例如内置麦克风、蓝牙耳机麦克风、USB 麦克风。
  • 输出设备：接收系统播放的音频数据，例如扬声器、听筒、有线耳机、蓝牙耳机、HDMI。

AudioDeviceInfo 用 isSource() 和 isSink() 表示设备方向：

  • isSource() == true：音频输入设备。
  • isSink() == true：音频输出设备。

一个物理设备可能同时包含输入和输出能力。例如带麦克风的有线耳机，耳机部分是输出，麦克风部分是输入。

2. 常用 API#

常见入口是 AudioManager：

AudioManager audioManager = getSystemService(AudioManager.class);

// 查询当前连接的输出设备
AudioDeviceInfo[] outputs =
        audioManager.getDevices(AudioManager.GET_DEVICES_OUTPUTS);

// 查询当前连接的输入设备
AudioDeviceInfo[] inputs =
        audioManager.getDevices(AudioManager.GET_DEVICES_INPUTS);

getDevices() 返回的是当前系统已连接、符合条件的设备列表，不代表设备一定适合当前业务场景。比如蓝牙耳机可能支持媒体播放，但通话场景还要看是否可作为通信设备使用。

音频设备类型#

1. 常见输出设备#

类型 说明 典型场景
TYPE_BUILTIN_SPEAKER 内置扬声器 外放音乐、免提
TYPE_BUILTIN_EARPIECE 听筒 电话、语音通话
TYPE_WIRED_HEADPHONES 不带麦克风的有线耳机 媒体播放
TYPE_WIRED_HEADSET 带麦克风的有线耳机 通话、录音、播放
TYPE_BLUETOOTH_A2DP 蓝牙媒体音频 音乐、视频
TYPE_BLUETOOTH_SCO 蓝牙通话音频 电话、VoIP
TYPE_BLE_HEADSET BLE 音频耳机 新版蓝牙音频播放或通话
TYPE_BLE_SPEAKER BLE 音频音箱 外放播放
TYPE_USB_DEVICE USB 音频设备 外接声卡、USB DAC
TYPE_USB_HEADSET USB 耳机 通话、播放、录音
TYPE_HDMI / TYPE_HDMI_ARC / TYPE_HDMI_EARC HDMI 音频输出 电视、功放、家庭影院

2. 常见输入设备#

类型 说明 典型场景
TYPE_BUILTIN_MIC 内置麦克风 录音、语音识别、通话
TYPE_WIRED_HEADSET 有线耳机麦克风 通话、录音
TYPE_BLUETOOTH_SCO 蓝牙通话麦克风 电话、VoIP
TYPE_USB_DEVICE / TYPE_USB_HEADSET USB 输入设备 外接麦克风、USB 耳机
TYPE_TELEPHONY 电话链路相关设备 系统电话场景
TYPE_REMOTE_SUBMIX 远端或虚拟混音设备 投屏、录屏、系统级混音

设备类型只是描述“这是什么设备”。实际能不能被当前应用使用，还取决于 Android 版本、权限、当前音频模式、设备连接状态和厂商实现。

音频模式类型#

AudioManager 的 mode 描述当前系统处于什么音频场景。它会影响路由、音量策略以及底层音频链路。

1. 常见模式#

模式 说明 应用建议
MODE_NORMAL 普通模式，没有响铃或通话 音乐、视频、普通播放默认使用
MODE_RINGTONE 响铃模式 通常由系统电话或系统响铃流程使用
MODE_IN_CALL 电话通话模式 普通应用不要使用，主要给系统电话应用
MODE_IN_COMMUNICATION 音视频通话或 VoIP 模式 语音聊天、视频会议常用
MODE_CALL_SCREENING 来电筛选模式 电话相关能力使用
MODE_CALL_REDIRECT 电话重定向模式 电话相关能力使用
MODE_COMMUNICATION_REDIRECT VoIP 音频重定向模式 通信音频被转移到其他设备时使用
MODE_ASSISTANT_CONVERSATION 助手对话模式 交互式语音助手场景

对普通应用来说，最常接触的是两个模式：

  • 播放音乐、短视频、提示音：保持 MODE_NORMAL。
  • 做语音通话、视频会议、实时对讲：进入 MODE_IN_COMMUNICATION，结束后恢复 MODE_NORMAL。

不要为了“让声音从听筒出来”随便设置 MODE_IN_CALL。这个模式涉及电话层和平台混音行为，普通应用误用后容易导致路由异常、音量异常，甚至影响其他应用。

通信设备选择#

1. Android 12 及以上#

Android 12（API 31）开始，通信场景推荐使用 setCommunicationDevice() 选择通话设备。它替代了部分旧的扬声器和蓝牙 SCO 控制方式。

示例：通话中切到扬声器。

if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) {
    AudioManager audioManager = getSystemService(AudioManager.class);

    audioManager.setMode(AudioManager.MODE_IN_COMMUNICATION);

    AudioDeviceInfo speaker = null;
    for (AudioDeviceInfo device : audioManager.getAvailableCommunicationDevices()) {
        if (device.getType() == AudioDeviceInfo.TYPE_BUILTIN_SPEAKER) {
            speaker = device;
            break;
        }
    }

    if (speaker != null) {
        boolean accepted = audioManager.setCommunicationDevice(speaker);
        if (!accepted) {
            // 设备不可用，或者系统拒绝本次路由请求。
        }
    }

    // 通话结束时恢复。
    audioManager.clearCommunicationDevice();
    audioManager.setMode(AudioManager.MODE_NORMAL);
}

这里有两个重点：

  • setCommunicationDevice() 只能选择适合通信场景的设备，输入设备通常由系统自动匹配。
  • 通话结束必须调用 clearCommunicationDevice()，并把 mode 恢复到 MODE_NORMAL。

2. Android 12 之前#

旧版本中常见做法是配合 MODE_IN_COMMUNICATION、setSpeakerphoneOn()、蓝牙 SCO 相关 API 做路由控制。但这些 API 在新版本中逐步被替代，兼容代码要按系统版本分支处理。

如果只是普通媒体播放，不应该主动控制通信设备。普通播放更应该通过 AudioAttributes 表达用途，让系统自己选择合适路由。

音频焦点#

音频焦点解决的是“多个应用同时想发声时，谁该继续播放、谁该暂停或降低音量”的问题。它不是音量大小，也不是输出设备，而是一套应用之间协作的播放仲裁机制。

1. 什么时候请求焦点#

一般规则是：准备真正开始播放前请求焦点，停止播放后释放焦点。

AudioAttributes attributes = new AudioAttributes.Builder()
        .setUsage(AudioAttributes.USAGE_MEDIA)
        .setContentType(AudioAttributes.CONTENT_TYPE_MUSIC)
        .build();

AudioFocusRequest focusRequest = new AudioFocusRequest.Builder(
        AudioManager.AUDIOFOCUS_GAIN)
        .setAudioAttributes(attributes)
        .setOnAudioFocusChangeListener(focusChange -> {
            switch (focusChange) {
                case AudioManager.AUDIOFOCUS_LOSS:
                    // 长时间失去焦点，应该暂停并等待用户重新触发播放。
                    break;
                case AudioManager.AUDIOFOCUS_LOSS_TRANSIENT:
                    // 短暂失去焦点，例如来电或语音播报，通常暂停。
                    break;
                case AudioManager.AUDIOFOCUS_LOSS_TRANSIENT_CAN_DUCK:
                    // 可以继续播放，但应该降低音量。
                    break;
                case AudioManager.AUDIOFOCUS_GAIN:
                    // 重新获得焦点，可以恢复音量或继续播放。
                    break;
            }
        })
        .build();

int result = audioManager.requestAudioFocus(focusRequest);
if (result == AudioManager.AUDIOFOCUS_REQUEST_GRANTED) {
    // 开始播放。
}

// 播放彻底结束后释放焦点。
audioManager.abandonAudioFocusRequest(focusRequest);

上面是 Android 8.0（API 26）及以上推荐的写法。更低版本需要使用旧的 requestAudioFocus(listener, streamType, durationHint) 接口。

2. 常见焦点类型#

类型 含义 典型场景
AUDIOFOCUS_GAIN 长时间获取焦点 音乐、播客、长视频
AUDIOFOCUS_GAIN_TRANSIENT 短时间获取焦点，希望对方暂停 语音播报、短音频
AUDIOFOCUS_GAIN_TRANSIENT_MAY_DUCK 短时间获取焦点，允许对方降低音量继续播 导航提示音、消息提示音
AUDIOFOCUS_GAIN_TRANSIENT_EXCLUSIVE 短时间独占焦点，不希望其他声音继续 录音、语音识别前的敏感场景

如果应用播放的是语音内容，例如有声书、课程、会议音频，不建议在失去焦点时简单 duck。语音内容被压低后可懂度会明显下降，更合理的策略通常是暂停，等焦点回来后再恢复。

3. Android 12 之后的变化#

Android 12（API 31）之后，系统会更强地介入音频焦点处理。比如媒体或游戏应用失去焦点时，系统可能强制淡出并静音。Android 15（API 35）开始，目标版本较高的应用还需要满足前台应用或前台服务等条件，才能成功请求音频焦点。

所以音频焦点不能只当成“礼貌回调”。在新系统上，它已经会直接影响应用是否能继续发声。

音量类型#

音量类型解决的是“用户按音量键时调的是哪一路音量”的问题。它和音频焦点、音频设备、音频模式都有关，但不是同一个概念。

1. 传统 stream 类型#

Android 早期主要通过 stream type 表示音量类型：

类型 说明 常见场景
STREAM_MUSIC 媒体音量 音乐、视频、游戏
STREAM_VOICE_CALL 通话音量 电话、VoIP 通话
STREAM_RING 铃声音量 来电铃声
STREAM_NOTIFICATION 通知音量 通知提示
STREAM_ALARM 闹钟音量 闹钟、倒计时提醒
STREAM_SYSTEM 系统音量 按键音、系统反馈
STREAM_DTMF 拨号音音量 电话拨号音
STREAM_ACCESSIBILITY 无障碍音量 屏幕阅读等无障碍反馈
STREAM_ASSISTANT 助手音量 语音助手反馈

可以通过 AudioManager 查询或调整指定 stream 的音量：

int current = audioManager.getStreamVolume(AudioManager.STREAM_MUSIC);
int max = audioManager.getStreamMaxVolume(AudioManager.STREAM_MUSIC);

audioManager.adjustStreamVolume(
        AudioManager.STREAM_MUSIC,
        AudioManager.ADJUST_RAISE,
        AudioManager.FLAG_SHOW_UI);

不过在播放器、提示音、通话这类具体播放对象上，更推荐使用 AudioAttributes 描述用途，而不是只依赖 stream type。

2. AudioAttributes 更适合描述用途#

AudioAttributes 用 usage 描述“为什么播放”，用 content type 描述“播放的是什么”。系统可以据此做更准确的路由、音量和焦点策略。

常见 usage 对应关系如下：

usage 含义 常见场景
USAGE_MEDIA 媒体内容 音乐、视频、播客
USAGE_GAME 游戏声音 游戏背景音、音效
USAGE_VOICE_COMMUNICATION 语音通信 VoIP、视频会议
USAGE_VOICE_COMMUNICATION_SIGNALLING 通话信令音 忙音、DTMF
USAGE_ALARM 闹钟 闹钟、强提醒
USAGE_NOTIFICATION 通知 普通通知提示
USAGE_NOTIFICATION_RINGTONE 铃声 来电铃声
USAGE_ASSISTANCE_NAVIGATION_GUIDANCE 导航引导 地图导航播报
USAGE_ASSISTANCE_SONIFICATION UI 声音反馈 按键音、操作反馈
USAGE_ASSISTANT 语音助手 助手回答和提示

例如普通音乐播放：

AudioAttributes mediaAttributes = new AudioAttributes.Builder()
        .setUsage(AudioAttributes.USAGE_MEDIA)
        .setContentType(AudioAttributes.CONTENT_TYPE_MUSIC)
        .build();

例如 VoIP 通话：

AudioAttributes callAttributes = new AudioAttributes.Builder()
        .setUsage(AudioAttributes.USAGE_VOICE_COMMUNICATION)
        .setContentType(AudioAttributes.CONTENT_TYPE_SPEECH)
        .build();

选择正确的 usage，比手动猜测 stream type 更稳。比如通话类音频使用 USAGE_VOICE_COMMUNICATION，系统才更容易匹配通信路由、通话音量和焦点策略。

3. 音量键到底调哪一路#

用户按音量键时，系统会根据当前前台播放、当前 mode、Activity 的音量控制设置和系统策略决定调哪一路音量。

普通媒体播放页面可以显式指定音量键控制媒体音量：

setVolumeControlStream(AudioManager.STREAM_MUSIC);

但这只是 Activity 层面对音量键行为的提示，不等于修改播放器的音频用途，也不等于切换输出设备。真正的播放语义仍然应该放在 AudioAttributes 里。

常见坑点#

1. 设备类型不等于当前路由#

getDevices() 查到的是当前连接设备，不代表当前音频一定走这个设备。比如手机同时连接蓝牙耳机和有线耳机时，系统会根据策略选择一路输出。

如果要判断通信场景当前设备，Android 12 及以上优先看 getCommunicationDevice()。

2. 蓝牙 A2DP 和 SCO 不是一回事#

TYPE_BLUETOOTH_A2DP 更偏媒体播放，音质较好但延迟和麦克风链路不适合传统通话。

TYPE_BLUETOOTH_SCO 更偏通话，支持麦克风和低延迟通信链路，但音质通常不如 A2DP。

所以音乐播放器和语音通话应用对蓝牙设备的选择策略通常不同。

3. mode 是全局状态，不是播放器私有属性#

setMode() 影响的是系统音频策略，不是某个播放器实例的局部配置。设置后不恢复，可能影响后续播放、录音、通话路由。

比较稳妥的习惯是：

try {
    audioManager.setMode(AudioManager.MODE_IN_COMMUNICATION);
    // 开始通话或实时语音。
} finally {
    audioManager.clearCommunicationDevice();
    audioManager.setMode(AudioManager.MODE_NORMAL);
}

实际项目里还要结合 Activity、Service、通话状态和异常退出路径，确保资源一定释放。

4. 焦点、音量和路由不要混用#

音频焦点决定谁该播放，音量类型决定调哪一路音量，音频路由决定声音从哪个设备出入。三者相关，但不能互相替代。

例如“切到扬声器”不应该通过调大 STREAM_MUSIC 实现；“让导航提示音压低音乐”不应该手动修改系统媒体音量，而应该用合适的音频焦点类型和 AudioAttributes。

5. 录音需要权限#

使用麦克风录音时需要 RECORD_AUDIO 权限。即使能查询到输入设备，没有权限也不能直接采集音频数据。

<uses-permission android:name="android.permission.RECORD_AUDIO" />

小结#

Android 音频输入输出可以按五层理解：

  • 设备层：AudioDeviceInfo 描述输入、输出和设备类型。
  • 场景层：AudioManager 的 mode 描述普通播放、响铃、通话、VoIP 等场景。
  • 路由层：系统根据设备连接、音频模式、应用请求和平台策略决定声音从哪里进、从哪里出。
  • 焦点层：音频焦点决定多个应用竞争播放时谁该继续、暂停或降低音量。
  • 音量层：stream type 和 AudioAttributes 共同影响音量控制、播放用途和系统策略。

普通媒体播放尽量少干预系统路由，重点是设置正确的 AudioAttributes 和音频焦点；实时语音或视频通话才需要关注 MODE_IN_COMMUNICATION 和通信设备选择。遇到音频问题时，先确认当前 mode，再确认设备类型和实际路由，最后检查焦点与音量类型，排查会更清晰。
