概述#

OpenSpec 是一个面向 AI 编程助手的 SDD（Spec-Driven Development，规格驱动开发）工具。它解决的问题不是“让 AI 更快写代码”，而是让一次需求变更先沉淀为可检查、可追踪、可归档的规格资产，再进入实现。

本文基于本机验证的 OpenSpec CLI 1.3.1，包名为 @fission-ai/openspec。当前默认 schema 是 spec-driven，核心产物链路是：

proposal -> specs -> design -> tasks -> apply -> archive

可以把它理解成一条轻量级需求流水线：

  • proposal.md：为什么做、做什么、影响什么
  • specs/：系统应该表现出什么行为
  • design.md：准备怎么实现，关键技术取舍是什么
  • tasks.md：可执行、可勾选、可验证的任务清单
  • archive：实现完成后，把变更合并回主规格

解决什么问题#

只靠聊天记录驱动 AI 写代码，常见问题是需求散落、上下文漂移、实现和预期不一致。OpenSpec 用仓库内的 openspec/ 目录把需求、规格、设计和任务固定下来，让人和 AI 都围绕同一组文件工作。

初始化后的核心目录如下：

openspec/
  changes/
    archive/
    add-dark-mode/
      .openspec.yaml
      README.md
      proposal.md
      specs/
        ui-theme/
          spec.md
      design.md
      tasks.md
  specs/
    ui-theme/
      spec.md

其中 openspec/changes/ 存放进行中的变更，openspec/specs/ 存放已经生效的主规格，openspec/changes/archive/ 存放完成后的历史变更。

安装与初始化#

OpenSpec 需要 Node.js 20.19.0 或更高版本。

npm install -g @fission-ai/openspec@latest
openspec --version

在项目根目录初始化：

cd your-project
openspec init

如果要非交互式指定 AI 工具，可以使用 --tools：

openspec init --tools codex
openspec init --tools claude,cursor
openspec init --tools none

--tools all 会尝试为所有支持的工具生成指令文件。团队项目里建议只启用正在使用的工具，避免生成过多无关配置。

初始化后，如果升级了 OpenSpec 包，需要在项目内刷新指令文件：

npm install -g @fission-ai/openspec@latest
openspec update

日常工作流程#

完整工作流可以概括为“先探索，再生成规格，再实现归档”。日常使用时优先记住 /opsx 命令主线：

flowchart TD
    A["/opsx:explore
澄清需求"] --> B{规格生成方式}
    B -- 快速生成 --> C["/opsx:propose
生成 proposal/specs/design/tasks"]
    B -- 分步生成 --> D["/opsx:new
创建 change"]
    D --> E["/opsx:continue 或 /opsx:ff
补齐 artifact"]
    C --> F["openspec validate
校验规格"]
    E --> F
    F --> G{校验通过}
    G -- 否 --> H["修改 artifact"]
    H --> F
    G -- 是 --> I["/opsx:apply
按 tasks.md 实现"]
    I --> J["/opsx:verify
验证实现"]
    J --> K{实现符合规格}
    K -- 否 --> L["修正代码或规格"]
    L --> F
    K -- 是 --> M["/opsx:archive
归档并更新主规格"]

如果团队要求显式同步 delta specs，可以在 /opsx:archive 前增加 /opsx:sync；多数情况下归档命令已经覆盖主规格更新。

1. 探索问题#

需求还不清楚时，不要直接让 AI 生成代码，可以先进入探索：

/opsx:explore 增加深色模式，要求兼容现有主题配置

这个阶段适合让 AI 读代码、澄清边界、确认影响范围，但不应该改代码。

2. 创建变更#

常用快速路径是：

/opsx:propose add-dark-mode

它会创建一个 change，并尝试一次性生成 proposal.md、specs/、design.md、tasks.md。

如果改动比较复杂，更推荐分阶段推进：

/opsx:new add-dark-mode
/opsx:continue add-dark-mode
/opsx:continue add-dark-mode

new 只创建变更骨架，continue 会按 schema 继续补齐下一个 artifact。这样更适合需要人工审阅 proposal、spec，再继续设计和拆任务的场景。

3. 查看状态#

openspec status --change add-dark-mode

典型输出会显示 artifact 完成度：

Change: add-dark-mode
Schema: spec-driven
Progress: 2/4 artifacts complete

[x] proposal
[x] specs
[ ] design
[-] tasks (blocked by: design)

blocked 表示前置 artifact 还没完成。默认 schema 中，tasks 依赖 specs 和 design。

4. 校验规格#

实现前至少跑一次校验：

openspec validate add-dark-mode --type change

提交前建议跑全量校验：

openspec validate --all --strict

脚本或 CI 中可以使用 JSON 输出：

openspec validate --all --json --no-interactive

如果仓库规格很多，可以调整并发：

OPENSPEC_CONCURRENCY=12 openspec validate --all --strict

5. 执行实现#

/opsx:apply add-dark-mode

apply 会读取 tasks.md，按待办任务实现，并在完成后更新复选框。这里有一个关键约束：任务必须使用标准复选框格式，否则进度追踪可能失效。

## 1. Theme Infrastructure

- [ ] 1.1 Create theme context provider
- [ ] 1.2 Add theme persistence in localStorage

## 2. UI Integration

- [ ] 2.1 Add theme toggle button
- [ ] 2.2 Verify initial render follows saved theme

6. 验证与归档#

实现后先验证：

/opsx:verify add-dark-mode

确认实现、规格、设计和任务状态一致后归档：

/opsx:archive add-dark-mode

也可以使用 CLI 归档：

openspec archive add-dark-mode
openspec archive add-dark-mode --yes

归档会校验变更、合并 delta specs 到 openspec/specs/，并把 change 移动到 openspec/changes/archive/。

如果是纯工具、文档或基础设施改动，不需要更新主规格，可以明确跳过：

openspec archive update-ci-config --skip-specs

端到端示例#

假设要给系统增加深色模式，最小闭环可以这样走：

/opsx:propose add-dark-mode

生成的 proposal.md 应该聚焦动机和影响：

## Why

Users need a dark theme to reduce eye strain in low-light environments.

## What Changes

- Add a user-facing theme toggle.
- Persist the selected theme locally.
- Apply theme variables during initial render.

## Capabilities

### New Capabilities
- `ui-theme`: Theme selection and persistence behavior

### Modified Capabilities

## Impact

- Frontend theme provider
- Header toolbar
- CSS variables

对应的 delta spec 应该描述行为，而不是实现细节：

## ADDED Requirements

### Requirement: Theme selection persists
The system SHALL persist the user's selected theme across browser sessions.

#### Scenario: User selects dark theme
- **WHEN** the user switches from light theme to dark theme
- **THEN** the system applies dark theme styling
- **AND** the selection is restored after page reload

实现前先检查状态和校验：

openspec status --change add-dark-mode
openspec validate add-dark-mode --type change --strict

然后执行：

/opsx:apply add-dark-mode
/opsx:verify add-dark-mode
/opsx:archive add-dark-mode

这个流程的重点是：spec 写系统行为，design 写技术方案，tasks 写可执行步骤。不要把三者混在一个文件里。

常用 CLI 命令#

命令 用途
openspec init [path] 初始化 OpenSpec 目录和工具指令
openspec update [path] 更新项目中的 AI 指令文件
openspec list 查看进行中的 changes
openspec list --specs 查看主规格
openspec show <name> 查看 change 或 spec
openspec validate <name> 校验 change 或 spec
openspec status --change <id> 查看 artifact 完成状态
openspec instructions [artifact] --change <id> 输出生成 artifact 或执行 apply 所需的增强指令
openspec new change <name> 创建新变更目录
openspec archive <name> 归档完成的变更
openspec view 打开交互式 dashboard
openspec templates 查看当前 schema 使用的模板路径
openspec schemas 查看可用 workflow schema
openspec schema validate <name> 校验 schema
openspec completion install zsh 安装 shell 补全
openspec config list 查看全局配置

几个适合脚本化的命令：

openspec list --json
openspec show add-dark-mode --json
openspec validate --all --json --no-interactive
openspec status --change add-dark-mode --json
openspec templates --json

Slash Commands#

命令 场景
/opsx:explore 只讨论和澄清问题，不改代码
/opsx:propose <name> 一次性创建变更并生成主要 artifact
/opsx:new <name> 创建变更骨架
/opsx:continue <name> 继续补齐下一个 artifact
/opsx:ff <name> 快速生成实现所需的全部 artifact
/opsx:apply <name> 按 tasks.md 实现
/opsx:verify <name> 验证实现是否匹配规格
/opsx:sync <name> 将 delta specs 同步到主 specs
/opsx:archive <name> 归档完成变更
/opsx:bulk-archive 批量归档多个完成变更
/opsx:onboard 引导式学习完整流程

日常最小闭环通常是：

/opsx:propose -> /opsx:apply -> /opsx:verify -> /opsx:archive

复杂需求更适合：

/opsx:explore -> /opsx:new -> /opsx:continue -> /opsx:apply -> /opsx:verify -> /opsx:archive

配置说明#

全局配置文件可以这样查看：

openspec config path
openspec config list

macOS 和 Linux 默认路径通常是：

~/.config/openspec/config.json

常见配置项：

配置 说明
profile 工作流配置，常见值为 core 或 custom
delivery 指令交付方式，支持 both、skills、commands
workflows custom profile 下启用的 workflow 列表
featureFlags 实验特性开关

切换 profile：

openspec config profile
openspec config profile core

直接设置配置：

openspec config set delivery commands
openspec config get delivery
openspec config unset delivery
openspec config reset

OpenSpec 会读取环境变量控制部分行为：

export OPENSPEC_TELEMETRY=0
export DO_NOT_TRACK=1
export OPENSPEC_CONCURRENCY=12
export NO_COLOR=1

其中 OPENSPEC_TELEMETRY=0 和 DO_NOT_TRACK=1 可用于关闭匿名使用统计。

Schema 与模板#

当前内置 schema 可以这样查看：

openspec schemas
openspec schema which spec-driven
openspec templates --schema spec-driven

spec-driven 的 artifact 定义是：

Artifact 生成文件 依赖
proposal proposal.md 无
specs specs/**/*.md proposal
design design.md proposal
tasks tasks.md specs、design

如果团队需要自定义流程，可以 fork 内置 schema：

openspec schema fork spec-driven team-workflow
openspec schema validate team-workflow

schema 命令当前属于实验能力，适合在团队内部统一后再使用。个人项目通常不用改 schema，只需要遵守默认模板即可。

规格写法约束#

specs/<capability>/spec.md 是 OpenSpec 最重要的契约文件。默认 schema 要求使用 delta 标题：

## ADDED Requirements
## MODIFIED Requirements
## REMOVED Requirements
## RENAMED Requirements

每个 requirement 使用三级标题：

### Requirement: User can export data
The system SHALL allow users to export their data in CSV format.

每个 scenario 使用四级标题：

#### Scenario: Successful export
- **WHEN** user clicks "Export"
- **THEN** system downloads a CSV file

需要特别注意：

  • 需求描述优先使用 SHALL 或 MUST，避免模糊的 should、may
  • 每个 requirement 至少要有一个 scenario
  • scenario 必须使用 #### Scenario，不要写成三级标题或普通列表
  • MODIFIED Requirements 必须包含修改后的完整 requirement block
  • REMOVED Requirements 应写清楚删除原因和迁移方式

团队落地建议#

1. 把 OpenSpec 变更作为 PR 的入口#

一个较稳的 PR 结构是：

代码变更
openspec/changes/<change-name>/
测试结果

评审时先看 proposal.md 和 specs/，确认行为契约没有偏，再看实现是否符合 design.md 和 tasks.md。

2. 给 CI 增加规格校验#

如果项目使用 GitHub Actions，可以增加一个轻量校验：

name: openspec

on:
  pull_request:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20.19.0"
      - run: npm install -g @fission-ai/openspec@1.3.1
      - run: openspec validate --all --strict --no-interactive

如果希望总是使用最新版，可以把安装命令改成 @latest；如果团队更看重可复现，建议固定版本。

3. 归档前做三项检查#

归档前至少确认：

  • openspec validate --all --strict 通过
  • tasks.md 中实现任务已经完成
  • specs/ 描述的是用户可观察行为，而不是代码实现细节

归档不是清理文件，而是把变更纳入长期规格。不要把未确认的临时方案归档进主规格。

常见坑#

1. 把实现细节写进 proposal#

proposal.md 主要回答为什么要做、做什么、影响哪些能力。具体技术方案应该放到 design.md。

2. tasks.md 不是标准复选框#

apply 阶段会解析 tasks.md。如果任务不是 - [ ] 格式，就可能无法追踪进度。

3. MODIFIED Requirements 写得太短#

修改已有规格时，如果只写新增的一两句话，归档时可能丢失原有细节。稳妥做法是复制完整 requirement，再写更新后的完整版本。

4. 跳过 validate 直接 archive#

归档会影响主规格。除非非常明确，否则不要使用 --no-validate。纯文档或工具变更可以用 --skip-specs，但仍建议保留校验。

5. 忽略版本差异#

OpenSpec 仍在快速演进，官方文档、main 分支和本地 CLI 版本可能存在差异。写团队文档时建议标明版本，并以本地命令输出为准：

openspec --version
openspec --help
openspec config list
openspec schemas

小结#

OpenSpec 的价值不在于多生成几份 Markdown，而是让 AI 开发过程有稳定的中间产物：变更动机、行为规格、技术设计和任务清单。小改动可以走 /opsx:propose -> /opsx:apply -> /opsx:archive，复杂改动则用 /opsx:explore、/opsx:new 和 /opsx:continue 分阶段补齐规格。

对团队项目来说，建议把 openspec/changes/ 当成需求评审和实现评审的共同入口，把 openspec/specs/ 当成长期维护的行为契约。这样 AI 不再只跟着聊天上下文走，而是围绕仓库里可审阅、可校验、可归档的规格工作。
