概述#

MySQL 是常见的关系型数据库，主要用来存储和查询结构化数据。

这篇文章聚焦最基础也最常用的内容，包括连接数据库、建库建表、表结构修改、基本 CRUD，以及主键、索引、事务和常见坑点。

连接数据库#

最常见的进入方式是命令行连接：

mysql -u root -p

如果是连接指定主机和端口：

mysql -h 127.0.0.1 -P 3306 -u root -p

连接成功后，可以先看有哪些数据库：

SHOW DATABASES;

退出客户端：

exit

建库建表#

创建一个测试库：

CREATE DATABASE demo_db;

切换到这个库：

USE demo_db;

查看当前库里的表：

SHOW TABLES;

下面建一张最简单的用户表：

CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50) NOT NULL,
  age INT,
  city VARCHAR(50),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

这里有几个关键点：

1. 字段类型#

  • INT：整数
  • VARCHAR(50)：最长 50 个字符的字符串
  • DATETIME：日期时间

2. 约束#

  • PRIMARY KEY：主键，唯一标识一条记录
  • AUTO_INCREMENT：自增，适合主键
  • NOT NULL：不能为空
  • DEFAULT：默认值

可以用下面的命令查看表结构：

DESC users;

再补几组开发里常见的表操作。

1. 新增字段#

ALTER TABLE users
ADD COLUMN email VARCHAR(100);

2. 修改字段类型或约束#

ALTER TABLE users
MODIFY COLUMN name VARCHAR(100) NOT NULL;

3. 重命名字段#

ALTER TABLE users
CHANGE COLUMN city user_city VARCHAR(50);

4. 删除字段#

ALTER TABLE users
DROP COLUMN email;

5. 添加和删除索引#

ALTER TABLE users
ADD INDEX idx_name (name);

ALTER TABLE users
DROP INDEX idx_name;

如果要删除测试表或测试库：

DROP TABLE users;
DROP DATABASE demo_db;

上面这些 ALTER TABLE 示例用于展示常见改表动作，实际执行时建议一次改一个点，并先用 DESC users; 或 SHOW CREATE TABLE users; 确认当前表结构。

基本 CRUD#

增删改查，也就是常说的 CRUD，是 MySQL 最基础的用法。

1. 插入数据#

INSERT INTO users (name, age, city)
VALUES ('Alice', 18, 'Shanghai');

一次插入多条：

INSERT INTO users (name, age, city)
VALUES
  ('Bob', 20, 'Beijing'),
  ('Cindy', 22, 'Shenzhen');

2. 查询数据#

查询全部数据：

SELECT * FROM users;

只查部分字段：

SELECT name, city FROM users;

带条件查询：

SELECT * FROM users WHERE age >= 20;

排序：

SELECT * FROM users ORDER BY age DESC;

限制返回条数：

SELECT * FROM users LIMIT 2;

分页查询：

SELECT * FROM users LIMIT 0, 10;

组合使用：

SELECT name, age
FROM users
WHERE city = 'Beijing'
ORDER BY age DESC
LIMIT 5;

3. 更新数据#

UPDATE users
SET age = 21
WHERE name = 'Bob';

4. 删除数据#

删除指定记录：

DELETE FROM users
WHERE name = 'Alice';

5. 一条完整练习链路#

如果你是第一次上手，可以按下面这条链路直接练一遍：

CREATE DATABASE demo_db;
USE demo_db;

CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50) NOT NULL,
  age INT,
  city VARCHAR(50),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO users (name, age, city)
VALUES
  ('Alice', 18, 'Shanghai'),
  ('Bob', 20, 'Beijing');

SELECT * FROM users;

UPDATE users
SET age = 21
WHERE name = 'Bob';

DELETE FROM users
WHERE name = 'Alice';

SELECT * FROM users;

常用查询条件#

实际开发里，查询通常不会只写一个 =。

1. 比较运算#

SELECT * FROM users WHERE age > 18;
SELECT * FROM users WHERE age <= 30;

2. 区间与集合#

SELECT * FROM users WHERE age BETWEEN 18 AND 25;
SELECT * FROM users WHERE city IN ('Beijing', 'Shanghai');

排除条件：

SELECT * FROM users WHERE city NOT IN ('Beijing', 'Shanghai');

3. 模糊匹配#

SELECT * FROM users WHERE name LIKE 'A%';

这里 % 表示任意长度的字符。

如果只匹配单个字符，可以用 _：

SELECT * FROM users WHERE name LIKE '_o%';

4. 空值判断#

SELECT * FROM users WHERE city IS NULL;

判断空值不能写成 city = NULL，这是初学者最常见的错误之一。

主键、索引、事务分别解决什么问题#

1. 主键#

主键用来唯一标识一条记录，解决的是“怎么准确区分每一行数据”的问题。

比如 users 表里的 id：

id INT PRIMARY KEY AUTO_INCREMENT

它的价值主要有两点：

  • 保证每条记录都有唯一身份
  • 方便更新、删除、关联查询时精确定位数据

如果没有主键，表里的记录虽然也能存，但后续维护和定位会很混乱。

2. 索引#

索引的作用可以简单理解为“加快查找”。

比如经常按 name 查询时，可以加一个索引：

CREATE INDEX idx_name ON users(name);

这样做的好处是查询更快，但也有代价：

  • 插入和更新会稍微慢一点
  • 索引会占用额外空间
  • 不是索引越多越好

所以索引要建在高频查询、高频过滤的字段上，而不是看到字段就加。

例如下面这类查询更可能用到索引：

SELECT * FROM users WHERE name = 'Bob';

但如果你对字段做了函数处理，索引可能就不容易命中：

SELECT * FROM users WHERE UPPER(name) = 'BOB';

3. 事务#

事务适合处理“一组操作必须同时成功”的场景，比如转账。

START TRANSACTION;

UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;

COMMIT;

如果中间某一步失败，应该回滚：

ROLLBACK;

事务的重点不是语法，而是场景意识：只要多个 SQL 之间存在强一致性要求，就要考虑事务。

常见坑点#

1. UPDATE 和 DELETE 忘记写 WHERE#

下面这种写法会影响整张表：

UPDATE users SET age = 18;
DELETE FROM users;

如果不是你想要的结果，后果通常很严重。

2. NULL 判断写错#

错误写法：

SELECT * FROM users WHERE city = NULL;

正确写法：

SELECT * FROM users WHERE city IS NULL;

3. WHERE 条件写得不准确#

例如你本来只想更新一个城市的数据，却把条件写得太宽：

UPDATE users
SET age = 25
WHERE city = 'Beijing';

这条 SQL 可能会改到很多行。正式执行前，最好先用同样条件跑一遍查询：

SELECT * FROM users WHERE city = 'Beijing';

4. 以为建了索引就一定快#

如果查询条件不走索引，或者数据量本身很小，索引未必能带来明显收益。索引优化要结合具体 SQL 和执行计划看。

5. SELECT * 用得太随意#

开发和排查问题时，SELECT * 很方便，但在正式查询里不一定合适：

  • 会把不需要的字段也查出来
  • 表字段变多后，返回内容会越来越重
  • 不利于后续接口和 SQL 的稳定维护

能明确字段时，优先写成下面这样：

SELECT id, name, city FROM users WHERE id = 1;

6. 改表前不先看结构#

在执行 ALTER TABLE 前，最好先确认当前表结构，否则很容易重复加字段，或者把字段类型改错。

DESC users;
SHOW CREATE TABLE users;

小结#

先把建库、建表、改表、CRUD 这几组操作练熟，再补上主键、索引和事务的使用场景，MySQL 的基础就够用了。真正容易出问题的地方通常不是语法不会写，而是 WHERE 条件写宽、NULL 判断写错、改表前不确认结构，以及过度依赖索引。
