概述#

Redis 是常见的内存数据库，通常用来做缓存、计数、排行榜、分布式锁等高频读写场景。

这篇文章聚焦最基础也最常用的内容，包括连接 Redis、常见数据类型、基本读写操作、过期时间，以及实战里常见的使用坑点。

连接 Redis#

最常见的方式是使用命令行客户端：

redis-cli

如果要连接指定主机和端口：

redis-cli -h 127.0.0.1 -p 6379

如果服务设置了密码：

redis-cli -h 127.0.0.1 -p 6379 -a your_password

连接成功后，可以先做一个简单检查：

PING

返回 PONG 说明 Redis 服务可用。

基本概念#

Redis 的数据以 key -> value 的形式存储。和关系型数据库不同，它没有表结构，重点是根据业务场景选择合适的数据类型。

开发里最常见的几类数据：

  1. String：最常用，适合缓存字符串、数字、JSON
  2. Hash：适合存对象结构
  3. List：适合消息队列、按顺序保存数据
  4. Set：适合去重
  5. ZSet：适合排行榜

String 常用操作#

String 是 Redis 使用最频繁的数据类型。

1. 写入和读取#

SET user:1001:name "alice"
GET user:1001:name

2. 覆盖写入#

SET user:1001:name "bob"

SET 会直接覆盖旧值，这一点要特别注意。

3. 数字自增#

SET page:view 100
INCR page:view
INCRBY page:view 10

这类操作适合做访问量、点赞数、库存计数等场景。

4. 删除 key#

DEL user:1001:name

Hash 常用操作#

Hash 适合存一组相关字段，比如用户信息。

1. 写入字段#

HSET user:1001 name "alice" age 18 city "Shanghai"

2. 读取字段#

HGET user:1001 name
HGETALL user:1001

3. 更新字段#

HSET user:1001 age 20

4. 删除字段#

HDEL user:1001 city

List 常用操作#

List 适合按顺序保存一批数据。

1. 左右插入#

LPUSH tasks "task1"
RPUSH tasks "task2"

2. 查看列表内容#

LRANGE tasks 0 -1

3. 弹出元素#

LPOP tasks
RPOP tasks

如果只是做最简单的队列，LPUSH + RPOP 或 RPUSH + LPOP 就够用了。

Set 与 ZSet 常用操作#

1. Set 去重#

SADD tags "mysql" "redis" "mysql"
SMEMBERS tags

Set 会自动去重，适合标签、用户 ID 集合、黑名单等场景。

2. ZSet 排行榜#

ZADD rank 100 user1 88 user2 95 user3
ZRANGE rank 0 -1 WITHSCORES
ZREVRANGE rank 0 2 WITHSCORES

ZSet 会按分数排序，适合排行榜、权重队列等场景。

过期时间#

Redis 很多场景都和“临时数据”有关，所以过期时间非常重要。

1. 设置过期时间#

SET login:code "123456"
EXPIRE login:code 60

也可以一次写入并设置过期时间：

SET session:token "abc123" EX 3600

2. 查看剩余过期时间#

TTL session:token

3. 取消过期时间#

PERSIST session:token

如果缓存、验证码、登录态没有设置过期时间，通常会留下很多无用数据。

key 与 value 的常见设计方式#

实战里，Redis 通常不是随便起一个 key、再随便塞一个 value，而是把“业务含义”和“数据形态”一起设计好。

1. key 要能直接定位目标数据#

常见写法是把业务类型、唯一标识、数据含义拼进 key 里：

user:1001
user:1001:profile
order:5001:status
rank:daily

这样代码拿到用户 id、订单 id、榜单类型后，就能直接拼出 key，而不是再做复杂查询。

2. value 可以是整个对象#

详情缓存最常见的做法是直接存对象 JSON：

user:1001 -> {"id":1001,"name":"alice","age":18}

3. value 也可以只是部分结果#

如果接口只需要少数字段，通常只缓存这部分数据：

user:1001:card -> {"name":"alice","avatar":"a.png"}

4. 简单状态适合存简单值#

很多场景只需要一个状态值或计数值：

order:5001:status -> PAID
article:2001:view_count -> 9821

5. 集合关系优先用 Redis 原生结构#

如果业务本身就是排行榜、标签集合、最近浏览记录，通常直接用 Redis 的数据类型表达：

rank:daily -> ZSet
user:1001:tags -> Set
user:1001:recent_views -> List

简单说，key 负责“怎么找到数据”，value 负责“数据用什么形式存”。设计得越清楚，后续维护越轻松。

常用管理操作#

查看当前库里的 key：

KEYS *

生产环境里更推荐渐进式扫描：

SCAN 0

切换数据库：

SELECT 1

查看 key 是否存在：

EXISTS user:1001

查看 key 的类型：

TYPE user:1001

原子性与实战理解#

Redis 单条命令通常具备原子性，这也是它很适合做计数器和简单状态控制的原因。

例如下面的自增操作：

INCR order:id

即使并发请求很多，这条命令本身也不会被拆开执行。

但要注意，多个命令拼起来不一定天然安全。例如先 GET，再在代码里计算，最后再 SET，中间就可能被别的请求改掉。

如果业务依赖“多步操作必须一起成功”，要考虑事务、Lua 脚本，或者重新设计数据更新方式。

常见坑点#

1. 直接把 Redis 当 MySQL 用#

Redis 适合高频读写、短数据路径、简单结构，不适合直接替代关系型数据库做复杂查询和强事务场景。

2. 忘记设置过期时间#

例如验证码、登录态、临时缓存如果没有过期时间，数据会一直堆积。

SET sms:code "123456" EX 300

3. KEYS * 用在生产环境#

KEYS * 很方便，但数据量大时会明显影响 Redis。排查线上问题时，优先用 SCAN。

4. 一个 key 存太大的 value#

例如把很大的 JSON、很长的列表直接塞进一个 key，会增加网络传输和内存压力，也会让排查问题更麻烦。

5. 误以为 Redis 天生不会丢数据#

Redis 默认以内存为主，持久化策略、主从复制和部署方式都会影响数据可靠性。缓存类数据问题不大，但核心业务数据不能只靠“放进 Redis 就安全”这种假设。

6. 不区分 key 命名#

如果 key 命名很乱，后期维护会很痛苦。实际开发里最好统一前缀，例如：

user:1:name
user:1:profile
order:1001:status

小结#

Redis 入门阶段，重点是先掌握常见数据类型和最基础的读写命令，再理解过期时间、原子性和 key 设计。真正容易出问题的地方通常不是命令不会写，而是数据类型选错、忘记设置过期时间、把大对象塞进单个 key，或者在线上滥用 KEYS *。
